| **Conversational business intelligence and operations platform.** Connect your systems, query your data, visualise it, and act on it — all from a single chat interface.  * * *  ## What it is  Matrix connects to the tools an organisation already uses — CRM, support, finance, file storage — and lets anyone query that data, visualise it, or trigger automated workflows in natural language. No SQL knowledge required. No switching between tools.  The interface is a chat. Every interaction, whether it returns a number, a chart, a pipeline that updates records, or a plain-language explanation, sta rts the same way: the user types a question or instruction.  > **Data never passes through the LLM.** Matrix generates a SQL query and wraps the response in natural language. The actual data is queried directly from your database and never touches the model. Raw data stays inside your infrastructure.  ![](https://t26433606.p.clickup-attachments.com/t26433606/0d2f5d1c-4c15-4310-a3aa-db988903552b/Matrix%20-%20high%20level.png)  User Interface mockup (=Nexus for Matrix):  [matrix-design (4).html](https://t26433606.p.clickup-attachments.com/t26433606/9681d5bb-43e4-4a54-b576-69ac14ac6ee2/matrix-design%20\\(4\\).html)  ## Architecture  Five layers, each with a single responsibility.  \| Layer \| Component \| Role \| \| ---\| ---\| --- \| \| Interface \| Nexus chat \| Receives messages, classifies intent, routes to the right capability \| \| Capability \| Query \| Generates SQL, runs it against PostgreSQL, returns results \| \| Capability \| Visualise \| Renders charts and dashboards from query results \| \| Capability \| Act \| Builds a structured action plan for user approval, then executes \| \| Pipeline \| Partic — read \| Ingests data from external systems into PostgreSQL on a schedule \| \| Pipeline \| Partic — write \| Executes approved write actions back to external systems \| \| Pipeline \| Biami \| Runs multi-step, conditional business processes \| \| Data \| PostgreSQL + storage \| Central store for all structured records, metadata, and files \| \| Control \| Approval gate \| Every write action requires explicit human confirmation \| \| Control \| Scheduler \| Drives both pipelines on a time-based schedule \| \| Control \| Audit log \| Records every query and write — read-only, always on \|  **Read and write pipelines are fully separate.** Different instances, different credentials, no shared code paths. A system can be connected read-only and write access enabled later as a deliberate separate step.  * * *  ## Spaces  A Space is the primary organisational unit. Each Space has its own dashboards, conversation history, tasks, scheduled jobs, and access controls.  The main area has three views:  *   **Chat** — full-screen Nexus interface; default landing when no dashboard is open *   **Dash** — saved dashboard widgets for this Space, with Nexus chat docked beneath it *   **Tasks** — running, queued, and completed tasks with subtask-level progress, engine tags, and Schedule / Pause / View log controls  Spaces are listed in the sidebar. Dashboards nest under their Space. Generic conversations appear under Recent chats. The sidebar footer links to **Integrations**, **Scheduled tasks**, and **Settings**.  * * *  ## Integrations  Each integration runs a read pipeline, a write pipeline, or both. Supported at launch: Salesforce, HubSpot, Zendesk, Intercom, NetSuite, QuickBooks, Google Workspace (Drive, Gmail, Calendar), and direct database connections.  When a file source is connected, Matrix extracts metadata — title, author, date, summary — and lands it in PostgreSQL alongside structured records, making documents queryable in the same conversation as CRM or support data.  * * *  ## Model selection  The composer exposes a per-message model picker:  *   **Claude Opus 4.8** — default; best for reasoning, complex actions, and planning *   **Claude Haiku 4.5** — fast and lower cost; appropriate for simple data queries *   **Local (on-prem)** — shown only when an admin has configured a local endpoint in Settings; keeps even the response wrapper inside the organisation's infrastructure  * * *  ## User journey  ### 1 · Sign in and set up  New users sign in via email or SSO and are taken through a brief space setup wizard: name the space, invite teammates. All steps are skippable and available from Settings later. Returning users land directly in their last-used Space.  ### 2 · Connect data  From Integrations, the user selects a system, authorises it via OAuth or API key, and chooses whether to enable read, write, or both. Read pipelines begin syncing immediately. File sources extract metadata in the background. Sync status shows pipeline health, last sync time, and errors.  ### 3 · Query  The user asks a question in natural language. Nexus generates SQL, runs it against PostgreSQL, and returns the answer. The SQL is shown inline (collapsed). A provenance note confirms which sources were queried and that data stayed in the database. Follow-up questions refine the query conversationally.  ### 4 · Visualise  When the user asks to chart or visualise results, a chart renders inline with auto-selected chart type and controls to switch type. Any chart can be saved to the Space dashboard with a name. Saved charts appear immediately as widgets on the Dash view.  ### 5 · Act  When the user asks to do something — notify owners, update records, create tasks — Nexus generates a structured action plan and presents it in plain language. The plan shows affected records, specific operations, and the engine that will execute (Partic or Biami). The user dry-runs, approves, or discards. Approval triggers execution; the task appears in the Tasks tab.  ### 6 · Schedule  Any completed task can be scheduled from its card using the Schedule button. Scheduled jobs appear in the global Scheduled tasks screen accessible from the sidebar, where they can be paused, edited, or removed. Scheduled read syncs keep data fresh; scheduled write actions automate recurring operational work — all with a full audit trail.  ### 7 · Day-to-day use  The user opens their Space, lands on Dash, and asks questions in the docked Nexus composer scoped to that dashboard's data. Approved actions progress in Tasks. New visualisations are saved in one click. Scheduled jobs run in the background.  * * *  ## Design principles  **Data privacy is architectural.** The LLM never sees raw records — only a result summary to wrap. The local model option extends this to on-prem inference for maximum sensitivity.  **Read and write are always separate.** Different pipelines, different credentials, independently audited.  **The agent proposes; the human approves.** No write executes without an explicit confirmation. The approval gate is not optional.  **Chat is the shell.** Query, visualise, act, plan, schedule — all from the same interface. The chat classifies intent and routes it; users never navigate to a separate tool.  **Everything is audited.** Every query and write is recorded with timestamp, user, engine, and enough context for rollback.
